修订变更的已有规划制品并保持它们之间的一致性。只编辑规划文件，绝不编辑代码。

**输入**: 可选指定变更名称（如 `/har:update add-oauth`）。如省略，从对话上下文推断或列出可用变更。

**步骤**

1. **选择变更**

   如果未提供名称，列出 `.claude/har/changes/`（排除 archive/）中最近修改的变更让用户选择。标注最近修改的为 "(推荐)"。

   **重要**: 不要猜测或自动选择，始终让用户选择。

2. **读取已有制品**

   列出变更目录下的所有已有文件并全部读取。

3. **理解修订请求**

   - 如果用户指定了具体修订（"design 改用 JWT"），以此为起始编辑
   - 如果用户只说"更新"/"整理一致性"，则进行一致性审查：检查所有已有制品之间的矛盾、缺口、重复

4. **读取并协调**

   - 读取修订涉及的制品以及变更的其他已有制品
   - 应用请求的编辑后，检查每个其他已有制品 — **任意方向**：修改后期制品可能要求修订前期制品，反之亦然
   - 记录所有不一致、缺失、矛盾之处
   - 仅修改已有文件。不创建不存在的制品 — 如果发现缺失制品，建议用 `/har:plan` 继续生成
   - 如果变更已一致，直接告知不做修改

5. **确认后逐个写入**

   - 展示每个拟修订内容和理由，用户确认后再写入
   - 如果用户拒绝某个修订，不写入该文件
   - 如果需要大幅重写，参考对应制品的模板格式

6. **提示下一步（仅建议，不自动执行）**

   - 仍有缺失制品 → 建议 `/har:plan`
   - 已实现但规划已修订 → 建议 `/har:apply` 将差异带入代码
   - 全部完成 → 建议 `/har:archive`

**输出示例**

```
## 修订: add-oauth

### 拟修订内容

1. **design.md** — 将 Token 存储从 Session 改为 JWT
   理由：JWT 支持无状态验证，与分布式部署更匹配
   
2. **tasks.md** — 新增"JWT 密钥轮换"任务
   理由：design 中增加了 JWT 安全要求

确认以上修订？
```

**护栏**
- 只编辑规划制品 — 绝不编辑实现代码。如果修订意味着代码变更，指向 `/har:apply`
- 仅修改已有文件，不创建新制品
- 每个编辑确认后再写入
- 如果修订改变了变更的核心意图而非精炼它，建议用 `/har:plan` 创建新变更
