按照变更规划的任务清单逐项实现代码。

**输入**: 可选指定变更名称（如 `/har:apply add-oauth`）。如省略，尝试从对话上下文推断。如果模糊不清，列出可用变更让用户选择。

**步骤**

1. **选择变更**

   如果用户提供了名称，直接使用。否则：
   - 从对话上下文中推断
   - 如果只有一个活跃变更，自动选择
   - 如果模糊，列出 `.claude/har/changes/`（排除 archive/）让用户选择

   明确告知："使用变更: <name>" 以及如何覆盖。

2. **读取上下文**

   读取变更目录下的所有制品文件获取完整上下文：
   - `proposal.md` — 了解动机和范围
   - `design.md` — 了解技术方案（如果存在）
   - `specs/` — 了解需求规格（如果存在）
   - `tasks.md` — 任务清单

3. **显示当前进度**

   解析 tasks.md 中的 checkbox：
   - 已完成：`- [x]` 的数量
   - 待完成：`- [ ]` 的数量

   显示：
   ```
   ## 实现: <change-name>
   **进度:** N/M 任务完成

   ### 待完成任务
   - [ ] X.Y 任务描述
   ...
   ```

4. **逐项实现任务（循环直到完成或暂停）**

   对每个待完成的任务：
   - 显示正在处理的任务
   - 进行所需的代码修改
   - 保持变更最小化和聚焦
   - 完成后将 tasks.md 中的 `- [ ]` 改为 `- [x]`
   - 继续下一个任务

   **暂停条件：**
   - 任务描述不清晰 → 请求澄清
   - 实现过程中发现设计问题 → 建议用 `/har:update` 修订制品
   - 遇到错误或阻塞 → 报告并等待指导
   - 用户中断

5. **完成或暂停时显示状态**

   显示：
   - 本次会话完成的任务
   - 总体进度："N/M 任务完成"
   - 如果全部完成 → 建议用 `/har:archive` 归档
   - 如果暂停 → 说明原因并等待指导

**输出示例 — 实现中**

```
## 实现: add-oauth

处理任务 3/7: 实现 OAuth 授权端点
[...实现过程...]
✓ 任务完成

处理任务 4/7: 添加 Token 刷新逻辑
[...实现过程...]
✓ 任务完成
```

**输出示例 — 实现完成**

```
## 实现完成

**变更:** add-oauth
**进度:** 7/7 任务完成 ✓

### 本次完成
- [x] 1.1 创建 OAuth 配置结构
- [x] 1.2 实现授权端点
...

所有任务完成！可以用 `/har:archive` 归档此变更。
```

**输出示例 — 实现暂停**

```
## 实现暂停

**变更:** add-oauth
**进度:** 4/7 任务完成

### 遇到的问题
Token 刷新逻辑中，第三方 API 返回的错误格式与文档不一致。

**选项:**
1. 按实际返回格式适配
2. 联系第三方确认文档
3. 其他方案

你想怎么做？
```

**护栏**
- 持续执行直到完成或阻塞
- 实现前先读取所有制品获取上下文
- 任务模糊时暂停并询问，不猜测
- 发现设计问题时暂停并建议修订制品
- 保持代码变更小且聚焦每个任务
- 完成任务后立即更新 checkbox
- 遇到错误、阻塞或模糊需求时暂停
